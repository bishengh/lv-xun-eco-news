## 1. 引言

### 1.1 编写目的

本文档是"绿讯 — 全国环保新闻聚合平台"的项目设计报告，详细描述系统的架构设计、数据库设计、各微服务的内部设计、前端设计、部署方案以及关键技术决策，为开发实施和后续维护提供参考。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **关注点分离** | 三层架构（数据层/业务层/表现层）各司其职，微服务独立部署 |
| **异步优先** | API 采用 async/await 异步模型，数据库使用异步驱动 asyncpg |
| **数据幂等** | 爬虫写入采用 UPSERT 机制，重复抓取不会产生脏数据 |
| **容错降级** | 爬虫提供 fallback 预存数据，单来源失败不影响整体运行 |
| **配置外置** | 所有关键参数（数据库连接、调度时间、抓取参数）通过环境变量配置 |
| **容器化** | 全部服务通过 Docker 容器化，Docker Compose 编排 |

### 3.3 JSONB 字段设计

`scrape_reports.source_details` 存储每个来源的抓取明细，结构如下：

```json
[
  {
    "source_id": "mee",
    "source_name": "生态环境部",
    "items_found": 45,
    "items_new": 3,
    "status": "success",
    "duration": 1.56,
    "error": null
  },
  {
    "source_id": "bj",
    "source_name": "北京",
    "items_found": 0,
    "items_new": 0,
    "status": "failed",
    "duration": 20.5,
    "error": "无法抓取到数据"
  }
]
```

选择 JSONB 的原因：
- 各来源的详细数据结构灵活，适合半结构化存储
- 避免为每次抓取生成 35 条关联记录，一次查询即可获取完整报告
- PostgreSQL JSONB 支持索引和查询操作符

## 4. 爬虫服务设计

### 4.1 模块结构

```
scraper-service/
├── scheduler.py          # 定时调度入口
├── scraper.py            # 核心抓取逻辑
├── db_writer.py          # 数据库写入模块

```

### 4.2 调度器设计（scheduler.py）

```
启动
  │
  ├── 注册信号处理（SIGTERM/SIGINT → 优雅退出）
  │
  ├── [可选] RUN_ON_START=true → 立即执行一次
  │
  └── BlockingScheduler
        └── CronTrigger(hour=8, minute=0, timezone=Asia/Shanghai)
              └── run_scraper()
                    └── scraper.main()
```

关键配置：
- `misfire_grace_time=3600`：容忍 1 小时的延迟触发
- `max_instances=1`：同一时刻最多运行 1 个抓取实例
- `PYTHONUNBUFFERED=1`：确保 Docker 日志实时输出

### 4.5 容错机制

| 场景 | 处理方式 |
|------|----------|
| 网络超时 | 重试 3 次，间隔 2 秒 |
| WAF 412 拦截 | 切换 Session 模式（建立 cookie） |
| HTTP 非 200 | 记录警告，跳过当前页面 |
| 抓取不足 3 条 | 使用 fallback 预存数据 |
| 异常崩溃 | 使用 fallback 数据 + 记录日志 |
| DB 写入失败 | 回滚事务 + 记录 db_error |
| 单来源失败 | 不影响其他来源，继续执行 |

### 5.3 新闻筛选查询设计

`news_repo.get_filtered()` 支持多维度动态条件组合：

```python
async def get_filtered(db, region, source_id, category, keyword,
                       date_start, date_end, page, page_size):
    query = select(News).options(selectinload(News.source))

    if region:
        query = query.join(Source).where(Source.region == region)
    if source_id:
        query = query.where(News.source_id == source_id)
    if category:
        query = query.where(News.category == category)
    if keyword:
        query = query.where(News.title.ilike(f"%{keyword}%"))
    if date_start:
        query = query.where(News.date >= date_start)
    if date_end:
        query = query.where(News.date <= date_end)

    # 分页
    total = await db.execute(select(func.count()).select_from(query.subquery()))
    items = await db.execute(
        query.order_by(desc(News.date), desc(News.created_at))
             .offset((page - 1) * page_size)
             .limit(page_size)
    )
    return items, total, page, page_size
```

## 8. 关键设计决策

### 8.1 为什么选择微服务而非单体？

| 考量 | 单体 | 微服务（当前选择） |
|------|------|-------------------|
| 部署灵活性 | 整体部署 | 各服务独立升级 |
| 爬虫与API耦合 | 高耦合 | 完全解耦 |
| 技术栈自由 | 受限 | 爬虫 Python、API FastAPI、前端 React |
| 故障隔离 | 爬虫崩溃影响 API | 爬虫崩溃不影响用户访问 |
| 资源管理 | 统一 | 可独立伸缩 |

### 8.2 为什么选择 PostgreSQL？

| 特性 | SQLite | MySQL | PostgreSQL（当前选择） |
|------|--------|-------|----------------------|
| 数组字段（tags） | 不支持 | 不支持 | TEXT[] 原生支持 |
| JSONB | 不支持 | JSON（无索引） | JSONB + GIN 索引 |
| 全文搜索 | 不支持 | 有限 | to_tsvector + GIN |
| UPSERT | 有限 | 有限 | ON CONFLICT DO UPDATE |
| 异步驱动 | 不支持 | 有限 | asyncpg（高性能） |
| 枚举类型 | 不支持 | ENUM | CREATE TYPE |

### 8.3 为什么选择 FastAPI？

| 特性 | Flask | Django | FastAPI（当前选择） |
|------|-------|--------|-------------------|
| 异步支持 | 不原生 | 3.1+ 有限 | 原生 async/await |
| 自动文档 | 无 | 无 | Swagger + ReDoc |
| 类型校验 | 手动 | 手动 | Pydantic 自动 |
| 性能 | 较低 | 中等 | 接近 Go/Rust |
| 学习曲线 | 低 | 高 | 中 |

### 8.4 为什么使用 APScheduler 而非 cron？

| 考量 | 系统 cron | APScheduler（当前选择） |
|------|-----------|----------------------|
| Docker 兼容 | 需额外配置 | 纯 Python，无外部依赖 |
| 时区处理 | 依赖系统 | 原生 timezone 支持 |
| 错过执行补偿 | 无 | misfire_grace_time |
| 并发控制 | 无 | max_instances 限制 |
| 日志集成 | 分离 | 统一 Python logging |

### 8.5 UPSERT 策略

```sql
INSERT INTO news (id, title, url, ...)
VALUES (%s, %s, %s, ...)
ON CONFLICT (id) DO UPDATE SET
    title = EXCLUDED.title,
    summary = EXCLUDED.summary,
    date = EXCLUDED.date,
    category = EXCLUDED.category,
    tags = EXCLUDED.tags,
    updated_at = NOW()
RETURNING (xmax = 0) AS is_new
```

- `ON CONFLICT (id)`：以新闻 ID 为冲突检测键
- `DO UPDATE SET`：更新标题、摘要等可能变化的字段
- `RETURNING (xmax = 0)`：利用 PostgreSQL 内部行版本号判断是新增（`xmax=0`）还是更新
- 保证幂等：相同新闻多次抓取不会产生重复记录
