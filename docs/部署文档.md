#### 2.2.2 建表 SQL

```sql
-- 来源分类枚举
CREATE TYPE source_category AS ENUM ('national', 'province', 'municipality', 'autonomous');

-- 数据来源表
CREATE TABLE IF NOT EXISTS sources (
    id          VARCHAR(10)      PRIMARY KEY,
    name        VARCHAR(100)     NOT NULL,
    short_name  VARCHAR(20)      NOT NULL,
    region      VARCHAR(20)      NOT NULL,
    url         VARCHAR(300)     NOT NULL,
    category    source_category  NOT NULL,
    created_at  TIMESTAMPTZ      DEFAULT NOW()
);

-- 新闻表
CREATE TABLE IF NOT EXISTS news (
    id          VARCHAR(64)      PRIMARY KEY,
    title       TEXT             NOT NULL,
    url         TEXT             NOT NULL UNIQUE,
    date        DATE,
    summary     TEXT,
    source_id   VARCHAR(10)      NOT NULL REFERENCES sources(id) ON DELETE CASCADE,
    category    VARCHAR(50)      DEFAULT '环保资讯',
    tags        TEXT[]           DEFAULT '{}',
    merged      BOOLEAN          DEFAULT FALSE,
    view_count  INTEGER          DEFAULT 0,
    created_at  TIMESTAMPTZ      DEFAULT NOW(),
    updated_at  TIMESTAMPTZ      DEFAULT NOW()
);

-- 爬取日志表
CREATE TABLE IF NOT EXISTS scrape_logs (
    id              SERIAL          PRIMARY KEY,
    source_id       VARCHAR(10)     REFERENCES sources(id) ON DELETE CASCADE,
    started_at      TIMESTAMPTZ,
    ended_at        TIMESTAMPTZ,
    items_found     INTEGER         DEFAULT 0,
    items_new       INTEGER         DEFAULT 0,
    status          VARCHAR(20)     DEFAULT 'pending',
    error_message   TEXT
);

-- 抓取报告表
CREATE TABLE IF NOT EXISTS scrape_reports (
    id              SERIAL          PRIMARY KEY,
    report_date     DATE            NOT NULL,
    started_at      TIMESTAMPTZ     NOT NULL,
    ended_at        TIMESTAMPTZ,
    total_sources   INTEGER         DEFAULT 0,
    success_sources INTEGER         DEFAULT 0,
    failed_sources  INTEGER         DEFAULT 0,
    total_items     INTEGER         DEFAULT 0,
    new_items       INTEGER         DEFAULT 0,
    duration_seconds FLOAT          DEFAULT 0,
    source_details  JSONB           DEFAULT '[]'::jsonb,
    status          VARCHAR(20)     DEFAULT 'running',
    error_message   TEXT,
    created_at      TIMESTAMPTZ     DEFAULT NOW()
);
```

#### 2.2.5 RLS 安全策略

所有 4 张表均已启用 RLS，共 9 条策略：

| 表名 | 策略名 | 角色 | 操作 | 说明 |
|------|--------|------|------|------|
| sources | `sources_public_read` | anon | SELECT | 公开只读 |
| sources | `sources_service_all` | service_role | ALL | 服务端完全访问 |
| news | `news_public_read` | anon | SELECT | 公开只读 |
| news | `news_anon_update_viewcount` | anon | UPDATE | 允许更新浏览量 |
| news | `news_service_all` | service_role | ALL | 服务端完全访问 |
| scrape_logs | `scrape_logs_public_read` | anon | SELECT | 公开只读 |
| scrape_logs | `scrape_logs_service_all` | service_role | ALL | 服务端完全访问 |
| scrape_reports | `scrape_reports_public_read` | anon | SELECT | 公开只读 |
| scrape_reports | `scrape_reports_service_all` | service_role | ALL | 服务端完全访问 |

**安全设计原则**：
- `anon` 角色（前端用户）：仅读取 + news 浏览量更新
- `service_role` 角色（爬虫服务）：完全 CRUD 权限
- 所有 RPC 函数使用 `SECURITY DEFINER`，绕过 RLS 执行

#### 2.3.3 Edge Function 核心逻辑

```
请求 → CORS 预检处理 → URL 路径解析 → 路由匹配
                                         │
                    ┌────────────────────┼────────────────────┐
                    ▼                    ▼                    ▼
               /news/*              /sources/*           /reports/*
                    │                    │                    │
              supabase.rpc()     supabase.from()     supabase.from()
              (调用 RPC 函数)     (直接查询)          (直接查询)
                    │                    │                    │
                    └────────────────────┼────────────────────┘
                                         ▼
                                   JSON Response
                              (统一 CORS headers)
```

**CORS 配置**：
```
Access-Control-Allow-Origin: *
Access-Control-Allow-Headers: authorization, x-client-info, apikey, content-type
Access-Control-Allow-Methods: GET, POST, PATCH, OPTIONS
```

### 2.4 密钥管理

| 密钥 | 类型 | 用途 | 安全级别 |
|------|------|------|----------|
| `anon` key (JWT) | Legacy | 前端公开 API 访问 | 公开（配合 RLS） |
| `publishable` key | Modern | 新应用推荐使用 | 公开（配合 RLS） |
| `service_role` key | Secret | 爬虫服务写入数据 | **机密**，仅服务端使用 |

> **安全警告**：`service_role` key 拥有绕过 RLS 的完全权限，严禁暴露在前端代码中。

## 5. 数据迁移

### 5.1 本地 → Supabase 迁移步骤

当需要将本地 Docker 数据库迁移到 Supabase 时：

```bash
# 1. 从本地 PostgreSQL 导出新闻数据
python3 -c "
import psycopg2, json
conn = psycopg2.connect(host='localhost', port=5432, dbname='lvxun', user='lvxun', password='lvxun_2026')
cur = conn.cursor()
cur.execute('SELECT id, title, url, date, summary, source_id, category, tags, merged, view_count, created_at, updated_at FROM news')
rows = cur.fetchall()
news = [{'id':r[0], 'title':r[1], 'url':r[2], 'date':r[3].isoformat() if r[3] else None, 'summary':r[4], 'source_id':r[5], 'category':r[6], 'tags':r[7] or [], 'merged':r[8], 'view_count':r[9], 'created_at':r[10].isoformat() if r[10] else None, 'updated_at':r[11].isoformat() if r[11] else None} for r in rows]
with open('/tmp/news_export.json', 'w') as f:
    json.dump(news, f, ensure_ascii=False)
print(f'Exported {len(news)} items')
conn.close()
"

# 2. 通过 Supabase REST API 批量导入
python3 -c "
import json, urllib.request
SUPABASE_URL = 'https://eeabqxxrqqvvlchvgeay.supabase.co'
ANON_KEY = 'your-anon-key'
with open('/tmp/news_export.json') as f:
    news = json.load(f)
for i in range(0, len(news), 100):
    batch = news[i:i+100]
    # 确保所有对象键一致（PostgREST 要求）
    payload = [{'id':n['id'], 'title':n['title'], 'url':n['url'], 'date':n.get('date'), 'summary':n.get('summary'), 'source_id':n['source_id'], 'category':n.get('category','环保资讯'), 'tags':n.get('tags',[]), 'merged':n.get('merged',False), 'view_count':n.get('view_count',0), 'created_at':n.get('created_at'), 'updated_at':n.get('updated_at')} for n in batch]
    data = json.dumps(payload, ensure_ascii=False).encode('utf-8')
    req = urllib.request.Request(f'{SUPABASE_URL}/rest/v1/news', data=data, method='POST', headers={'apikey':ANON_KEY, 'Authorization':f'Bearer {ANON_KEY}', 'Content-Type':'application/json', 'Prefer':'resolution=ignore-duplicates,return=minimal'})
    urllib.request.urlopen(req)
    print(f'Batch {i//100+1} done')
"
```

### 5.2 注意事项

- PostgREST 批量插入要求**所有对象键必须一致**，缺失字段用 `null` 填充
- 使用 `Prefer: resolution=ignore-duplicates` 跳过主键冲突
- 先插入 `sources`，再插入 `news`（外键约束）
- `scrape_reports.source_details` 为 JSONB 字段，导入时需确保是 JSON 对象而非字符串

## 7. 故障排除

### 7.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Edge Function 返回 500 | RPC 函数错误或数据库连接问题 | 查看 Edge Function Logs，检查 SQL 函数 |
| 前端 CORS 错误 | Edge Function CORS 头缺失 | 检查 Edge Function 中的 `corsHeaders` 配置 |
| 数据导入失败 `PGRST102` | PostgREST 批量插入对象键不一致 | 确保所有对象包含完全相同的键集合 |
| 新闻列表为空 | sources 表未导入或 RLS 策略缺失 | 检查 sources 数据和 `*_public_read` 策略 |
| 浏览量不增长 | `news_anon_update_viewcount` 策略缺失 | 添加 anon UPDATE 策略 |
| 爬虫无法写入 Supabase | 未使用 `service_role` key | 爬虫的 DATABASE_URL 需使用 Supabase 连接串 |
| Docker 日志为空 | Python stdout 缓冲 | 添加 `PYTHONUNBUFFERED=1` 环境变量 |
| APScheduler 时区错误 | TZ 未设置 | `docker-compose.yml` 中设置 `TZ=Asia/Shanghai` |

### 7.2 重置 Supabase 数据库

如需完全重建：

```sql
-- 1. 删除所有表（按依赖顺序）
DROP TABLE IF EXISTS scrape_reports CASCADE;
DROP TABLE IF EXISTS scrape_logs CASCADE;
DROP TABLE IF EXISTS news CASCADE;
DROP TABLE IF EXISTS sources CASCADE;
DROP TYPE IF EXISTS source_category CASCADE;

-- 2. 删除所有 RPC 函数
DROP FUNCTION IF EXISTS get_news_stats();
DROP FUNCTION IF EXISTS search_news(TEXT, TEXT, TEXT, TEXT, DATE, DATE, INT, INT);
DROP FUNCTION IF EXISTS get_latest_news(INT);
DROP FUNCTION IF EXISTS get_hot_news(INT);
DROP FUNCTION IF EXISTS get_news_detail(TEXT);
DROP FUNCTION IF EXISTS get_related_news(TEXT, INT);
DROP FUNCTION IF EXISTS get_sources_stats();
DROP FUNCTION IF EXISTS increment_view_count(TEXT);

-- 3. 按照 2.2 节重新执行建表、索引、RPC 函数、RLS 策略
-- 4. 按照 5.1 节重新导入数据
```
