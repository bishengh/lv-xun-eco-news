## 1. 引言

### 1.1 编写目的

本文档是"绿讯 — 全国环保新闻聚合平台"的软件需求规格说明书，旨在明确系统的功能需求、非功能需求、数据需求及接口需求，为系统设计、开发、测试和验收提供依据。

### 1.2 项目背景

我国生态环境信息分散于生态环境部及各省、自治区、直辖市、特别行政区的环保官方网站，用户获取全国环保动态需逐一访问 35 个不同网站，信息获取效率低、覆盖面不全。"绿讯"平台通过自动化抓取、智能去重合并、分类标签提取等技术手段，构建全国环保新闻一站式聚合平台，提高环保信息的可达性和使用效率。

## 2. 总体描述

### 2.1 产品概述

绿讯是一个基于微服务架构的全国环保新闻聚合平台，支持双模式部署：

- **生产环境（Supabase Cloud + Vercel）**：
  - **数据层**：Supabase PostgreSQL 15，RLS 安全策略，8 个 RPC 函数
  - **业务层**：Supabase Edge Function (Deno)，12 个 API 端点
  - **表现层**：React SPA，Vercel CDN 托管（`https://web-ten-nu-48.vercel.app`）

- **开发环境（Docker Compose）**：
  - **数据层**：PostgreSQL 16 关系数据库，结构化存储新闻、来源、抓取报告等数据
  - **业务层**：FastAPI 异步 API 服务，提供 RESTful 接口
  - **表现层**：React 单页面应用，Vite 开发服务器

系统通过定时调度器每日自动抓取全国 35 个环保数据源的最新新闻，经去重、分类、标签提取后存入数据库，并通过 Web 界面提供多维度的新闻检索和浏览服务。

### 2.2 用户特征

| 用户角色 | 描述 | 使用场景 |
|----------|------|----------|
| 公众用户 | 关注环保新闻的普通公众 | 浏览新闻列表、查看详情、按地区/分类筛选 |
| 环保从业者 | 环保部门工作人员、研究人员 | 检索特定地区/主题的环保政策和动态 |
| 系统管理员 | 平台运维人员 | 查看抓取报告、监控系统运行状态 |

## 3. 功能需求

### 3.1 新闻数据抓取（FR-01）

#### FR-01.1 全量数据源覆盖

- 系统**应**覆盖生态环境部及全国 34 个省、自治区、直辖市、特别行政区共 35 个环保数据来源
- 系统**应**针对不同网站结构提供多种解析策略（不少于 7 种）
- 系统**应**对 WAF 拦截、JS 渲染等异常情况提供 fallback 机制

#### FR-01.2 定时自动抓取

- 系统**应**每天早上 08:00（北京时间）自动执行全量数据抓取
- 调度时间**应**可通过环境变量配置
- 系统**应**支持启动时可选立即执行一次抓取（通过 `RUN_ON_START` 环境变量控制）
- 系统**应**支持手动触发抓取

#### FR-01.3 数据处理

- 系统**应**对抓取的新闻进行日期过滤，默认保留近 60 天的新闻
- 系统**应**对跨来源的重复新闻进行标题去重合并
- 系统**应**自动对新闻进行分类，覆盖不少于 9 个类别
- 系统**应**基于关键词自动提取新闻标签
- 系统**应**使用 UPSERT 机制写入数据库，保证数据幂等性

#### FR-01.4 抓取报告生成

- 每次抓取**应**自动生成一份抓取报告
- 报告**应**包含：日期、开始/结束时间、总耗时、成功/失败来源数、总抓取条数、新增条数
- 报告**应**包含每个来源的详细信息：来源名称、抓取条数、新增条数、耗时、状态、错误信息

### 3.2 新闻浏览（FR-02）

#### FR-02.1 首页展示

- 系统**应**在首页展示平台核心统计数据（来源数、新闻总数、今日更新数）
- 系统**应**在首页展示最新抓取报告摘要（成功率、新增数、耗时）
- 系统**应**展示最新新闻列表（默认 8 条）
- 系统**应**展示热门新闻列表（按浏览量排序，默认 6 条）
- 系统**应**提供按省份/地区快速浏览的入口

#### FR-02.2 新闻列表

- 系统**应**支持新闻分页展示（默认每页 20 条）
- 系统**应**支持以下筛选维度的任意组合：
  - 地区筛选
  - 数据来源筛选
  - 新闻分类筛选
  - 关键词搜索
  - 日期范围筛选
- 系统**应**按新闻日期倒序排列

#### FR-02.3 新闻详情

- 系统**应**展示新闻的标题、来源、日期、分类、标签
- 系统**应**提供新闻原文链接的跳转
- 系统**应**展示相关推荐新闻
- 系统**应**记录并展示新闻浏览量（每次访问详情自动 +1）

### 3.3 抓取报告查看（FR-03）

#### FR-03.1 报告列表

- 系统**应**提供抓取报告的分页列表
- 列表**应**展示：日期、状态、成功率、抓取条数、新增条数、耗时

#### FR-03.2 报告详情

- 系统**应**提供报告的详细查看页面
- 详情**应**展示核心指标面板：成功率、抓取条数、新增入库数、失败来源数、耗时
- 详情**应**分组展示成功来源和失败来源
- 成功来源**应**展示：来源名称、抓取条数、新增条数、耗时
- 失败来源**应**展示：来源名称、错误信息

### 3.4 数据来源管理（FR-04）

- 系统**应**提供数据来源的查询接口
- 系统**应**提供各来源的新闻数量统计

## 4. 非功能需求

### 4.1 性能需求（NFR-01）

| 指标 | 要求 |
|------|------|
| API 响应时间 | 常规查询 < 500ms |
| 单次全量抓取耗时 | < 10 分钟 |
| 数据库连接池 | 最小 10，最大 30 |
| 前端首屏加载 | < 3 秒（生产环境） |
| 并发用户支持 | >= 100 |

### 4.2 可用性需求（NFR-02）

- 系统**应**支持 `restart: unless-stopped` 策略，容器异常退出后自动重启
- 数据库**应**配置健康检查，依赖服务在数据库就绪后才启动
- 抓取调度器**应**配置 `misfire_grace_time`（1 小时），容忍短时间的延迟触发

### 4.3 可移植性需求（NFR-03）

- 系统**应**支持 Supabase Cloud + Vercel 云端部署（生产环境）
- 系统**应**通过 Docker Compose 实现一键本地部署（开发环境）
- 系统**应**支持 Linux、macOS、Windows 环境
- 所有配置**应**可通过环境变量覆盖

### 4.4 安全性需求（NFR-04）

- API **应**配置 CORS 策略，限制允许的源
- 数据库凭据**应**通过环境变量传递，不硬编码在代码中
- 云端数据库**应**启用 RLS（Row Level Security），区分 anon（公开只读）和 service_role（完全权限）
- 爬虫**应**设置合理的请求间隔（>= 0.5 秒），避免对目标网站造成压力

### 4.5 可维护性需求（NFR-05）

- 代码**应**遵循分层架构：Model → Repository → Service → Router
- 前端**应**遵循组件化开发，UI 组件与业务组件分离
- 系统**应**保留抓取日志（scrape_logs）和抓取报告（scrape_reports）以便排查问题

### 4.6 兼容性需求（NFR-06）

- 前端**应**采用响应式设计，适配桌面端（>= 1024px）、平板（>= 768px）、手机（>= 375px）
- 前端**应**兼容主流浏览器最近两个主要版本

## 6. 接口需求

### 6.1 外部接口

#### 6.1.1 数据源接口（出站 HTTP）

系统爬虫服务通过 HTTP/HTTPS 协议访问 35 个环保官方网站，抓取 HTML 页面内容。

- 协议：HTTP/HTTPS
- 方法：GET
- 超时：20 秒
- 重试：最多 3 次
- 请求间隔：>= 0.5 秒

### 6.2 内部接口（RESTful API）

#### 6.2.1 新闻接口

| 方法 | 路径 | 说明 | 参数 |
|------|------|------|------|
| GET | `/api/news` | 新闻列表 | region, source, category, keyword, date_start, date_end, page, pageSize |
| GET | `/api/news/latest` | 最新新闻 | count（默认 8） |
| GET | `/api/news/hot` | 热门新闻 | count（默认 6） |
| GET | `/api/news/stats` | 数据统计 | 无 |
| GET | `/api/news/{id}` | 新闻详情 | 无 |
| GET | `/api/news/{id}/related` | 相关推荐 | count（默认 5） |

#### 6.2.2 来源接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/sources` | 来源列表 |
| GET | `/api/sources/stats` | 来源统计 |
| GET | `/api/sources/{id}` | 来源详情 |

#### 6.2.3 报告接口

| 方法 | 路径 | 说明 | 参数 |
|------|------|------|------|
| GET | `/api/reports` | 报告列表 | page, page_size |
| GET | `/api/reports/latest` | 最新报告 | 无 |
| GET | `/api/reports/{id}` | 报告详情 | 无 |

#### 6.2.4 系统接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/health` | 健康检查 |

## 7. 约束与假设

### 7.1 约束

1. 系统仅抓取公开发布的环保新闻信息，不涉及非公开数据
2. 爬虫遵守目标网站的 robots.txt 协议（如有）
3. 系统不存储新闻全文内容，仅存储标题、摘要、原文链接等元数据
4. 系统运行于 Docker 环境，依赖 Docker Desktop 或兼容的容器运行时

### 7.2 假设

1. 目标网站的页面结构不会频繁大规模变更
2. 网络环境允许正常访问 35 个数据源网站
3. Docker 运行环境具备足够的计算资源（建议 >= 2GB RAM）
4. 用户使用支持 ES2020+ 的现代浏览器
